# Refactor Plan

## Automate release PDF upload via reusable docs-build workflow

- Date: 2026-03-22
- Status: Implemented
- Title: Extract reusable docs-build workflow and automate PDF upload to tag-triggered releases
- Kind: Refactor
- Severity: Low
- Stage: Shared
- Subsystem: CI / release management / documentation
- Modules:
  - `.github/workflows/docs-build.yml`
  - `.github/workflows/docs-publish.yml`
  - `.github/workflows/release.yml`

## Summary

The PDF reference manual (`refman.pdf`) was only uploaded to GitHub Releases via manual `workflow_dispatch` in
`docs-publish.yml`. Tag-triggered releases created by `release.yml` did not include it because the `release: published`
event fired by `github.token` does not trigger other workflows (GitHub Actions limitation).

This plan extracts the shared docs build job into a reusable `workflow_call` workflow (`docs-build.yml`) and has
`release.yml` call it in parallel with the platform dist builds, so `refman.pdf` is automatically included in every
tag-triggered release alongside the binary archives and checksums.

## Problem

1. `release.yml` creates the GitHub Release using `github.token`. Events generated by `GITHUB_TOKEN` do not trigger
   other workflows, so the `release: published` trigger in `docs-publish.yml` never fires for tag-triggered releases.
2. The docs build job (Python/uv/TeX setup, `gen-docs.sh --pdf --strict`, artifact uploads) was defined inline in
   `docs-publish.yml` with no way to reuse it from another workflow.
3. Duplicating the build steps in `release.yml` would create a maintenance burden.

## Solution

### Phase 1: Extract `docs-build.yml` as a reusable workflow

Create `.github/workflows/docs-build.yml` with `on: workflow_call`. Move the `build-docs` job from `docs-publish.yml`
into it, preserving all steps: checkout, Python 3.14 + uv setup, TeX toolchain install, `uv sync --group docs`,
`gen-docs.sh --pdf --strict`, Pages site staging, and artifact uploads (`github-pages`, `docs-markdown`, `docs-pdf`).

### Phase 2: Update `docs-publish.yml` to call the reusable workflow

Replace the inline `build-docs` job with `uses: ./.github/workflows/docs-build.yml`. All downstream jobs
(`deploy-pages`, `upload-release-pdf`, `sync-blog`) remain unchanged â€” they consume the same artifact names uploaded by
the reusable workflow.

### Phase 3: Call `docs-build.yml` from `release.yml`

Add a `build-docs` job to `release.yml` that calls `./.github/workflows/docs-build.yml`. This runs in parallel with
`build-dist`. Update `publish-release` to depend on both `build-dist` and `build-docs`, download the `docs-pdf` artifact
into `build/release-assets/`, and include `refman.pdf` in `SHA256SUMS` and the `gh release create` upload glob.

## Verification

1. Push a `v*` tag. The release workflow builds dist archives and docs in parallel. The published GitHub Release
   contains platform archives, `SHA256SUMS` (including `refman.pdf`), and `refman.pdf`.
2. Trigger `docs-publish.yml` via `workflow_dispatch`. The reusable `docs-build.yml` runs, and downstream jobs (Pages
   deploy, PDF upload, blog sync) work as before.
3. Trigger `docs-publish.yml` via `workflow_dispatch` with `upload_pdf_to_release=true` and a `release_tag`. The manual
   PDF upload path still works as a fallback.
