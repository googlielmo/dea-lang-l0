## Summary

Two workflow tests broke silently after recent refactors and were not caught because they are not part of the Stage 1 /
Stage 2 test gates that CI runs on every PR.

## Root Cause 1: Missing mock for `_source_url` in `test_dist_tools_lib_fallback.py`

`collect_stage2_build_provenance` gained a `_source_url` call which internally calls
`_git_output(repo_root, git_env, "remote", "get-url", "origin")`. The mock `side_effect` list in
`test_dist_tools_lib_fallback.py` only provided three return values (for `rev-parse HEAD`, `rev-parse --short HEAD`, and
`status --porcelain`), causing `StopIteration` on the fourth call.

### Fix

Add a fourth mock return value for the `git remote get-url origin` call.

## Root Cause 2: Stale VERSION-file assertions in `test_release_tag_policy.py`

The release and snapshot workflows no longer write a `VERSION` file inline with `printf 'dea/l0 ...' > VERSION`.
Instead, the version is passed via the `DEA_DIST_VERSION` environment variable and the dist tooling generates the
VERSION file. The test still asserted the old `printf` lines.

### Fix

Replace the `printf ... > VERSION` assertions with checks for the `DEA_DIST_VERSION` export lines that replaced them.

## Root Cause 3: Stale `git describe` assertion in `test_release_tag_policy.py`

The docs-publish workflow no longer uses `git describe --tags --abbrev=0 --match 'l0-v*'` to resolve the release tag. It
uses `github.event.release.tag_name` directly. The test still asserted the old `git describe` invocation.

### Fix

Replace the `git describe` assertion with a check for the `RELEASE_TAG: ${{ github.event.release.tag_name }}`
environment binding.

## Verification

`make test-workflows` passes on all platforms.
